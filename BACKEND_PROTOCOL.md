# XURA — Протокол взаимодействия клиент ↔ бэкенд

> Версия: 1.0  
> Дата: 2026-06-23  
> Сеть: XRPL Mainnet (`wss://xrplcluster.com`)  
> Источник истины: исходный код ветки `fix/random-generator`

---

## 1. Транспорт

Все игровые взаимодействия — это **XRP Payment-транзакции** в сети XRPL.  
Данные передаются в поле **MemoData** (UTF-8, закодировано в HEX uppercase).

```
MemoData = bytes(text, "utf-8").hex().upper()
```

Никаких HTTP-запросов между клиентом и игровым сервером нет — только on-chain транзакции.

## 3. Общий формат сообщений

### 3.1 Клиент → Сервер (ставка / заявка)

```
XRPL Payment:
  Destination: <адрес сервера>
  Amount:      <сумма в дропах>  (1 XRP = 1 000 000 drops)
  Memos:
    Memo:
      MemoData: HEX("<текст команды>")
```

### 3.2 Сервер → Клиент (ответ)

```
XRPL Payment:
  Destination: <адрес кошелька игрока>
  Amount:      <выплата в дропах>
  Memos:
    Memo:
      MemoData: HEX("<CMD>:<значение>")
```

Клиент принимает ответ через **WebSocket-подписку** на поток `transactions` для своего адреса.  
Декодирование: `bytes.fromhex(MemoData).decode("utf-8").upper()` — клиент всегда переводит в **uppercase**.

## 4. Игра «Угадай цвет» (Guess the Color)

### 4.1 Ставки

| Параметр | Значение |
|----------|---------|
| Адрес сервера | `rGrEJZaBFYhPGuyM7NiJbJw2yXVB9vJHah` |
| Минимальная ставка | **0.1 XRP** |
| Максимальная ставка | **100 XRP** |
| Точность суммы | до **6 знаков** после запятой |
| Множитель выплаты | **×2** |

### 4.2 Мемо ставки (клиент → сервер)

```
BET:RED:{referral}    — ставка на красный цвет
BET:BLK:{referral}    — ставка на чёрный цвет
```

| Поле | Значение |
|------|---------|
| `BET` | тип операции — игровая ставка |
| `RED` / `BLK` | выбранный цвет |
| `{referral}` | реферальный код игрока; `"0"` если реферала нет |

**Примеры:**
```
BET:RED:0              — ставка на красный, без реферала
BET:BLK:XURA99        — ставка на чёрный, реферал XURA99
```

### 4.3 Ожидаемые ответы сервера

| Ответ MemoData | Смысл | Выплата |
|---------------|-------|---------|
| `WIN:{число}` | Угадал цвет | `сумма_ставки × 2` XRP |
| `LOSE:{число}` | Не угадал | **12 дропов** (символически) |
| `JKPT:{число}` | Джекпот (клиент отображает как WIN) | сумма выплаты |
| `LOTO:{число}` | Лото-выигрыш (клиент отображает как WIN) | сумма выплаты |

- `{число}` — выпавшее число рулетки (0–36), которое клиент отображает на колесе.
- 0 — зелёное зеро, оба цвета (RED и BLK) **проигрывают** при выпадении нуля.
- Клиент ждёт транзакцию без таймаута; при LOSE всё равно должна прийти транзакция.

## 5. Игра «Угадай число» (Guess the Number)

### 5.1 Ставки

| Параметр | Значение |
|----------|---------|
| Адрес сервера | `rfcMxSEz4JP8zj65LU5Nw9hKfpEaD6Ss9` |
| Диапазон числа | **1–36** (0 не доступен для ставки) |
| Минимальная ставка | **0.1 XRP** |
| Максимальная ставка | **100 XRP** |
| Точность суммы | до **6 знаков** после запятой |
| Множитель выплаты | **×36** |

### 5.2 Мемо ставки (клиент → сервер)

```
BET:N:{число}:{referral}
```

| Поле | Значение |
|------|---------|
| `BET` | тип операции — игровая ставка |
| `N` | тип игры — число |
| `{число}` | выбранное число от 1 до 36 |
| `{referral}` | реферальный код; `"0"` если нет |

**Примеры:**
```
BET:N:7:0              — ставка на число 7, без реферала
BET:N:17:XURA99       — ставка на число 17, реферал XURA99
BET:N:36:ref5000      — ставка на число 36
```

### 5.3 Ожидаемые ответы сервера

| Ответ MemoData | Смысл | Выплата |
|---------------|-------|---------|
| `WIN:{число}` | Число угадано | `сумма_ставки × 36` XRP |
| `LOSE:{число}` | Число не угадано | **12 дропов** (символически) |
| `JKPT:{число}` | Джекпот (→ WIN) | сумма выплаты |
| `LOTO:{число}` | Лото (→ WIN) | сумма выплаты |

- `{число}` — выпавшее число (то, которое сгенерировал сервер). Клиент сравнивает с тем, что поставил игрок.
- Клиент не делает самостоятельного расчёта выигрыша — он доверяет `delivered_amount` из ответной транзакции.

## 6. Игра «Рулетка» (Roulette)

Полная спецификация: [`ROULETTE_BACKEND_SPEC.md`](./ROULETTE_BACKEND_SPEC.md)

Краткое резюме ниже.

### 6.1 Параметры

| Параметр | Значение |
|----------|---------|
| Адрес сервера | `rGrEJZaBFYhPGuyM7NiJbJw2yXVB9vJHah` |
| Мин. ставка на позицию | **0.1 XRP** |
| Макс. суммарная ставка | **100 XRP** |
| Тип рулетки | Европейская (0–36) |

### 6.2 Мемо ставки (клиент → сервер)

```
BET:R:{код1}@{сумма1},{код2}@{сумма2},...:{referral}
```

Итоговая сумма транзакции = сумма всех ставок в дропах.

**Примеры:**
```
BET:R:n17@1.5:0                          — прямая ставка на 17, 1.5 XRP
BET:R:r@2.0,b@2.0:XURA99               — красные + чёрные по 2 XRP
BET:R:n5@1.5,r@2.0,d1@0.5:ref123       — три ставки
```

### 6.3 Таблица кодов ставок рулетки

| Код | Ставка | Условие выигрыша | Множитель |
|-----|--------|-----------------|-----------|
| `n0` | Число 0 | выпало 0 | **×36** |
| `n1`–`n36` | Число 1–36 | выпало то число | **×36** |
| `r` | Красные | красное число | **×2** |
| `b` | Чёрные | чёрное число | **×2** |
| `o` | Нечётные | нечётное, ≠ 0 | **×2** |
| `e` | Чётные | чётное, ≠ 0 | **×2** |
| `l` | 1–18 | 1 ≤ число ≤ 18 | **×2** |
| `h` | 19–36 | 19 ≤ число ≤ 36 | **×2** |
| `d1` | 1-я дюжина | 1 ≤ число ≤ 12 | **×3** |
| `d2` | 2-я дюжина | 13 ≤ число ≤ 24 | **×3** |
| `d3` | 3-я дюжина | 25 ≤ число ≤ 36 | **×3** |
| `c1` | 1-я колонка | число % 3 == 1 | **×3** |
| `c2` | 2-я колонка | число % 3 == 2 | **×3** |
| `c3` | 3-я колонка | число % 3 == 0 | **×3** |

**Чёрные числа:** 2, 4, 6, 8, 10, 11, 13, 15, 17, 20, 22, 24, 26, 28, 29, 31, 33, 35  
**0** — зелёное зеро; все внешние ставки (r, b, o, e, l, h, d1–d3, c1–c3) проигрывают.

### 6.4 Ожидаемые ответы сервера

| Ответ MemoData | Смысл | Выплата |
|---------------|-------|---------|
| `WIN:{число}` | Хотя бы одна ставка выиграла | итоговая выплата по всем выигравшим позициям |
| `LOSE:{число}` | Все ставки проиграли | **12 дропов** |

## 7. Реферальная программа

### 7.1 Стать рефералом

| Параметр | Значение |
|----------|---------|
| Адрес сервера | `rGrEJZaBFYhPGuyM7NiJbJw2yXVB9vJHah` |
| Стоимость | **66 XRP** |
| Мемо | `REF` |

**Транзакция клиента:**
```
Destination: rGrEJZaBFYhPGuyM7NiJbJw2yXVB9vJHah
Amount:      66 000 000 drops  (66 XRP)
MemoData:    HEX("REF")        →  "524546"
```

**Ожидаемый ответ сервера:**
```
MemoData: HEX("REF:{присвоенный_код}")

Пример:
  REF:7483647    — сервер выдал реферальный код 7483647
  REF:XURA0001   — сервер выдал буквенно-цифровой код
```

Клиент сохраняет `{код}` и показывает его пользователю как его реферальный номер.  
Максимальное количество рефералов в системе: **666 666**.

---

### 7.2 Восстановление реферального статуса

| Параметр | Значение |
|----------|---------|
| Адрес сервера | `rGrEJZaBFYhPGuyM7NiJbJw2yXVB9vJHah` |
| Стоимость | **13 XRP** |
| Мемо | `REF:REC` |

**Транзакция клиента:**
```
Destination: rGrEJZaBFYhPGuyM7NiJbJw2yXVB9vJHah
Amount:      13 000 000 drops  (13 XRP)
MemoData:    HEX("REF:REC")    →  "5245463A524543"
```

**Ожидаемый ответ сервера:**
```
MemoData: HEX("REF:{восстановленный_код}")
```

Клиент ожидает тот же формат `REF:{код}`, что и при первичной регистрации.  
Сервер должен вернуть код, ранее привязанный к данному адресу кошелька.

## 8. Универсальный формат ответа сервера

Клиент разбирает **любой** ответ по одной схеме:

```
{CMD}:{значение}
```

| CMD | Действие клиента | `{значение}` |
|-----|-----------------|--------------|
| `WIN` | Показать выигрыш | выигрышное число (0–36) |
| `LOSE` | Показать проигрыш | выигрышное число (0–36) |
| `JKPT` | Показать выигрыш (джекпот) | выигрышное число |
| `LOTO` | Показать выигрыш (лото) | выигрышное число |
| `REF` | Сохранить реферальный код | сам код |

Клиент читает `delivered_amount` из `meta` ответной транзакции — это реальная сумма, которую получил игрок. Именно её клиент показывает в сообщении о выигрыше.

## 9. Как клиент принимает ответ

1. После отправки ставки клиент показывает экран Flasher (вращающееся колесо).
2. Клиент **подписан через WebSocket** на поток транзакций своего адреса (`accounts: [адрес_игрока]`).
3. При поступлении транзакции с `type: "transaction"` и полем `Memos`:
   - Клиент проверяет, что `Destination` — **не** адрес игрового сервера (т.е. это ответ от сервера, а не своя же ставка).
   - Декодирует `MemoData` из HEX в UTF-8, переводит в uppercase.
   - Парсит как `{CMD}:{значение}`.
4. Вызывает `Flasher.stopGame(text, win)` — останавливает колесо на `{значение}`.

```python
# Фильтр входящих транзакций на стороне клиента (упрощённо)
def should_process(event, my_address, server_addresses):
    tx   = event["transaction"]
    meta = event["meta"]
    return (
        event["type"] == "transaction"
        and tx.get("Destination") == my_address          # адресовано мне
        and tx.get("Account") not in server_addresses    # не от меня самого
        and meta.get("TransactionResult") == "tesSUCCESS"
        and "Memos" in tx
    )
```

> **Важно:** клиент фильтрует по признаку `Destination != SERVER_ADDRESS_GUESS_THE_COLOR`.  
> Поскольку три адреса — один, сервер должен отвечать транзакцией **с адреса сервера** на **адрес игрока**, чтобы клиент её обработал.

## 10. Кодирование MemoData

```python
def encode_memo(text: str) -> str:
    return text.encode("utf-8").hex().upper()

def decode_memo(hex_str: str) -> str:
    return bytes.fromhex(hex_str).decode("utf-8").upper()

# Примеры
encode_memo("BET:RED:0")       # → "4245543A5245443A30"
encode_memo("BET:N:17:XURA")   # → "4245543A4E3A31373A58555241"
encode_memo("BET:R:n5@1.5,r@2.0:0") # → "..."
encode_memo("WIN:17")           # → "57494E3A3137"
encode_memo("LOSE:0")           # → "4C4F53453A30"
encode_memo("REF")              # → "524546"
encode_memo("REF:REC")          # → "5245463A524543"
encode_memo("REF:7483647")      # → "5245463A37343833363437"
```

## 15. Резерв активации кошелька XRPL

XRPL требует **10 XRP резерва** для активации каждого аккаунта.  
Клиент **всегда вычитает 10 XRP** из полного баланса при отображении доступной суммы.

```java
// PaymentAndSocketManagerXRPL.getBalance()
доступный_баланс = полный_баланс − 10.000000 XRP
```

Если `полный_баланс ≤ 10 XRP` — клиент показывает `0.000000 XRP` и блокирует ставки.

**Что это значит для бэкенда:**
- Получив ставку от игрока, не удивляться, что на его адресе числится на 10 XRP больше, чем ставка.
- При расчёте «хватит ли у игрока средств» — помнить, что клиент уже вычел резерв, но XRPL-нода считает полный баланс включая резерв.
- Минимальный пополнение нового кошелька для начала игры: **> 10 XRP** (резерв) + сумма ставки + комиссия транзакции (~0.000012 XRP).

## 16. Обновление баланса по WebSocket

При каждой входящей транзакции клиент **параллельно** запускает два потока:
1. `NotifierRun` — разбирает мемо, показывает результат игры.
2. `UpdateBalanceRun` — обновляет отображаемый баланс.

`UpdateBalanceRun` читает итоговый баланс аккаунта **напрямую из события транзакции** (без отдельного запроса):

```json
event["meta"]["AffectedNodes"][1]["ModifiedNode"]["FinalFields"]["Balance"]
```

Значение в дропах, клиент переводит в XRP и вычитает резерв 10 XRP.

> **Важно для бэкенда:** структура `AffectedNodes` в XRPL не гарантирует порядок элементов.  
> Клиент обращается к индексу `[1]` — если сервер генерирует транзакции нестандартной структуры,  
> обновление баланса может упасть с `JSONException` (обработано через try/catch, не критично —  
> баланс обновится при следующем запросе).

## 17. Зарезервированные коды (не обрабатываются клиентом)

Следующие коды определены в приложении, но текущий парсер (`NotifierRun`) их **не обрабатывает** — они игнорируются:

| Код | Константа | Возможное назначение |
|-----|-----------|---------------------|
| `22` | `REFUND` | Возврат ставки (например, при ошибке сервера) |
| `99` | `DONATION` | Пожертвование / благодарность без игрового контекста |

Если сервер отправит мемо `REFUND:22` или `DONATION:99` — клиент получит транзакцию (баланс обновится), но никакого уведомления на экране **не появится**. Колесо продолжит крутиться до таймаута.

**Вывод:** эти коды зарезервированы для будущего или для внутренних нужд. Если бэкенд хочет сообщить об ошибке/возврате — на текущий момент нужно слать `LOSE:{число}` с нулевой суммой или использовать будущую версию клиента с поддержкой этих кодов.

## 18. Критические детали парсинга

### 18.1 Разбивка мемо на стороне клиента

```java
// NotifierRun.java
String[] parts = memoText.split(":", 2);   // лимит 2 — не более двух частей
String cmd          = parts[0];             // "WIN", "LOSE", "REF", ...
String serverNumber = parts.length > 1 ? parts[1] : "0";
```

`split(":", 2)` — **лимит 2**. Это означает:
- `"WIN:17"` → `["WIN", "17"]` ✓
- `"LOSE:0"` → `["LOSE", "0"]` ✓
- `"REF:7483647"` → `["REF", "7483647"]` ✓
- `"REF:abc:def"` → `["REF", "abc:def"]` — допустимо, клиент возьмёт `"abc:def"` целиком как код
- `"WIN"` (без двоеточия) → `["WIN"]` → `serverNumber = "0"` — колесо встанет на зеро

> **Правило:** всегда отправлять двоеточие и число. Мемо без числа = колесо на 0.

### 18.2 Uppercase

```java
String memoText = new String(hexToBytes(hexMemo), StandardCharsets.UTF_8).toUpperCase();
```

Клиент **принудительно переводит мемо в uppercase** перед switch.  
Сервер может отправлять мемо в любом регистре — `win:17`, `Win:17`, `WIN:17` — всё обработается одинаково.

### 18.3 Фильтр ответных транзакций

```java
// NotifierRun.run()
if (message.getJSONObject("transaction").has("Memos")
    && !message.getJSONObject("transaction").getString("Destination")
        .equals(StringEnum.SERVER_ADDRESS_GUESS_THE_COLOR.getValue())) {
    responseToBet(message);
}
```

Клиент обрабатывает транзакцию если:
1. В ней есть `Memos` — это ответ от сервера (а не какой-то входящий перевод без мемо).
2. `Destination` **не равен** `rGrEJZaBFYhPGuyM7NiJbJw2yXVB9vJHah` — чтобы не обработать собственную исходящую ставку повторно.

> **Важно:** клиент сравнивает `Destination` только с адресом сервера цвета/рулетки/рефералов.  
> Сервер «Угадай число» (`rfcMxSEz4JP8zj65LU5Nw9hKfpEaD6Ss9`) в фильтре не упомянут —  
> то есть ответы от него пройдут фильтр. Это работает корректно, поскольку  
> собственные ставки на «Угадай число» идут на другой адрес, и клиент их не получает обратно через подписку.

## 19. Таймлайн и идемпотентность

### 19.1 Время ответа

Клиент показывает крутящееся колесо **без таймаута** на стороне UI.  
Ограничение — только `LastLedgerSequence` в самой XRP-транзакции (~4 минуты).

Рекомендуемое время ответа сервера: **3–8 секунд** (имитирует спин рулетки).

### 19.2 Дедупликация

Каждая транзакция на XRPL имеет уникальный `hash`. Бэкенд обязан:

```
1. Получить входящую транзакцию
2. Проверить hash в базе обработанных — если уже есть, игнорировать
3. Записать hash как "в обработке"
4. Рассчитать результат и отправить ответ
5. Записать hash как "обработано"
```

### 19.3 Подписка бэкенда на XRPL WebSocket

```json
{
  "command": "subscribe",
  "streams": ["transactions"],
  "accounts": [
    "rGrEJZaBFYhPGuyM7NiJbJw2yXVB9vJHah",
    "rfcMxSEz4JP8zj65LU5Nw9hKfpEaD6Ss9"
  ]
}
```

Фильтр входящих ставок:
```python
def is_valid_bet(event):
    tx   = event["transaction"]
    meta = event["meta"]
    return (
        event.get("type") == "transaction"
        and meta.get("TransactionResult") == "tesSUCCESS"
        and "Memos" in tx
        and tx.get("Destination") in SERVER_ADDRESSES
    )
```
