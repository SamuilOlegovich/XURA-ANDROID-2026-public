# Flutter vs React Native — для портирования XURA на iOS

> Контекст: XURA — Java/Android, MVVM, Dagger Hilt, xrpl4j, 31 экран, 79 тестов.
> Цель: один кодовой базой покрыть Android + iOS.

---

## 🐦 Flutter (Google, язык Dart)

**Как работает:** рисует весь UI через собственный движок (Skia/Impeller) —
не использует нативные компоненты iOS/Android, рисует каждый пиксель сам.

### Плюсы
- Один код → Android + iOS (+ Web, Desktop при желании)
- Архитектура MVVM переносится 1:1 (BLoC / Riverpod ≈ ViewModel + LiveData)
- UI выглядит **одинаково** на обеих платформах — что нарисовал, то и увидишь
- Dart — строго типизированный, синтаксис очень близок к Java (2–3 недели на освоение)
- Hot reload — правишь код, видишь результат мгновенно
- Хорошие пакеты для кошелька:
  - `flutter_secure_storage` — замена Android Keystore / iOS Keychain
  - `local_auth` — биометрия (Touch ID / Face ID)
  - `mobile_scanner` — QR-сканирование
  - `web_socket_channel` — WebSocket к XRPL
- Отличная производительность (нет JS-моста, нативная компиляция)

### Минусы
- Dart — не Java, придётся учить (хотя порог низкий)
- UI **не нативный** — кнопки и анимации "флаттеровские", не стандартные iOS/Android
- Бинарник тяжелее (~15–20 МБ overhead от Flutter engine)
- XRPL-библиотека для Dart менее зрелая, чем xrpl4j или xrpl.js — часть логики
  придётся писать вручную (протокол JSON-RPC/WebSocket несложный, но время)

### Оценка переноса XURA
| Блок | Время |
|---|---|
| XRPL-клиент + WebSocket + бизнес-логика | 4–6 недель |
| UI (31 экран → Flutter виджеты) | 6–8 недель |
| Безопасность (Keystore → flutter_secure_storage) | 1 неделя |
| **Итого** | **~3.5 месяца** |

## ⚛️ React Native (Meta, язык TypeScript)

**Как работает:** компилирует JS-компоненты в **настоящие нативные** виджеты
iOS и Android. Кнопка на iOS — это реальная UIButton, не нарисованная.

### Плюсы
- **xrpl.js** — официальная библиотека от Ripple, самая зрелая и полная.
  Вся логика кошелька/транзакций уже написана и проверена сообществом.
  Это убирает самый рискованный кусок работы.
- UI выглядит **нативно** на каждой платформе — iOS как iOS, Android как Android
- TypeScript — строгая типизация, огромная экосистема npm
- Expo — надстройка над RN, убирает боль с Xcode/Gradle конфигурациями
- Богатая Web3/крипто экосистема: кошельки, подписи, шифрование — всё есть
- Пакеты для кошелька:
  - `react-native-keychain` — биометрия + безопасное хранилище
  - `vision-camera` — QR-сканирование
  - `@react-native-community/netinfo` — состояние сети
  - `react-native-quick-crypto` — нативная криптография (быстро)

### Минусы
- JavaScript/TypeScript — другая парадигма. Хуки (useState, useEffect) отличаются
  от LiveData. Нужно перестроить мышление.
- Производительность — JS-поток и нативный поток разделены. Сложные анимации
  и криптоoperации в JS могут лагать (решается нативными модулями)
- Отладка сложнее — два мира (JS + нативный), stacktrace иногда непонятные
- `node_modules` — исторически больное место, зависимости конфликтуют
- Новая архитектура (JSI + Fabric) лучше старого Bridge, но не все библиотеки
  успели обновиться

### Оценка переноса XURA
| Блок | Время |
|---|---|
| XRPL-клиент (xrpl.js уже готов!) | 1–2 недели |
| UI (31 экран → React компоненты) | 6–8 недель |
| Безопасность (react-native-keychain) | 1 неделя |
| **Итого** | **~3 месяца** |

## ⚖️ Прямое сравнение для XURA

| Критерий | Flutter | React Native |
|---|---|---|
| XRPL-библиотека | Слабее, частично самописная | **xrpl.js — официальная, зрелая** |
| Язык | Dart (близкий к Java) | TypeScript (другая парадигма) |
| UI нативность | Свой движок, одинаков везде | Настоящие нативные компоненты |
| Производительность | **Отличная** (нет JS bridge) | Хорошая (с JSI), хуже для крипто |
| Скорость старта | Средняя | **Быстрее** за счёт xrpl.js |
| Экосистема крипто/Web3 | Слабее | **Намного богаче** |
| Порог входа с Java | Низкий (Dart похож) | Средний (JS — другой мир) |
| Анимации / плавность | **Отличные** | Хорошие |
| Итоговое время | ~3.5 месяца | ~3 месяца |

## 🎯 Вывод

**React Native** — если хочешь быстрее и надёжнее в части XRPL.
xrpl.js написана Ripple, хорошо документирована, все операции с кошельком
уже там. Убирает самый рискованный кусок работы.

**Flutter** — если важнее контроль над UI и производительность.
Для кошелька с играми (рулетка, анимации выигрыша) Flutter даст
более плавный и предсказуемый результат на обеих платформах.

### Личная рекомендация
**React Native + Expo + TypeScript + xrpl.js** — xrpl.js экономит месяц
работы, а это решающий фактор для проекта одного разработчика.
Если в будущем захочется полноценного Web-приложения или DApp —
переход от xrpl.js к браузеру почти бесшовный.
